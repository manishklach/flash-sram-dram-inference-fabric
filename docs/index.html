Flash → DRAM → SRAM

Deterministic Inference Memory Fabric

A software-defined memory hierarchy that transforms low-cost NVMe flash into a hidden sequential streaming tier for AI inference

The Core Idea

Flash is not treated as slow RAM. Flash is treated as a high-capacity sequential stream source. DRAM is the predictive staging buffer. SRAM is the deterministic scratchpad hot path.

A synchronous flash read during token generation is a policy failure, not a normal cache miss. Model inference has highly predictable access patterns — transformer layers execute in order, weights can be linearized, KV cache has temperature and locality, MoE expert routing is often predictable. This predictability lets us use low-cost NVMe flash as a high-capacity sequential streaming tier, hidden behind DRAM staging and explicit SRAM scheduling.

The goal is not to make flash behave like true random-access memory. The goal is to avoid putting flash on the synchronous token path altogether.

Memory Hierarchy

Three tiers with distinct responsibilities, bandwidth characteristics, and cost profiles:

SRAM Scratchpad

  • 64 MB on-chip capacity
  • 5 ns latency, high bandwidth
  • Active token working set
  • Current attention tiles
  • Decompressed tensor tiles
  • Compute: SRAM hot path

DRAM / LPDDR Staging

  • 8-512 GB capacity
  • 80 ns latency, 100 GB/s
  • Predictive staging buffer
  • Warm KV blocks
  • Upcoming layer bundles
  • Residency metadata

NVMe Flash Capacity

  • 2-4 TB capacity
  • 80 μs latency (sequential)
  • 7 GB/s sequential bandwidth
  • Cold KV cache + MoE experts
  • Model weight bundles
  • Long-context history

The critical token path (SRAM → Compute → Token) must never include a synchronous flash read. Flash traffic happens exclusively in the hidden path (Flash → DRAM → SRAM), completing before compute needs the data.

Inference Pipeline

At any given decode step, three tiers operate concurrently on different parts of the model:

Token T, layer 12 is computing:

  SRAM:   serves active layer-12 tiles
  DRAM:   holds layers 13-16 and hot KV blocks
  Flash:  streams layers 17-32 and cold KV bundles into DRAM

By the time the model reaches layer 17, the next weight and KV data should already be present in DRAM and ready for promotion into SRAM. This is the core scheduling contract.

KV Cache Tiering

KV blocks are tagged with a temperature score based on recency, attention, sink token bonus, retrieval frequency, and reuse distance. The tiering policy places hot blocks in SRAM/DRAM, warm blocks in DRAM, and cold blocks in compressed flash.

MoE Expert Tiering

flowchart TD
    A[Router / Predictor] --> B[Top-K Likely Experts]
    B --> C[DRAM Prefetch]
    C --> D[Selected Expert Tiles]
    D --> E[SRAM]
    E --> F[Expert Compute]
    G[Cold Experts] --> H[SSD Flash]
    H --> C

MoE experts are prefetched into DRAM before the router selects them. Cold experts stay in compressed flash until predicted. Low routing entropy yields high prefetch accuracy.

Simulation Results

The simulator validates the architecture across 7 workloads × 3 interface modes × 5 prefetch policies. Key findings:

Stream-to-Scratchpad eliminates synchronous flash failures

Across all workloads, stream-to-scratchpad mode achieves 0 synchronous flash failures at the cost of some prefetch waste during cold-start windows. RAM-emulation mode produces 248-844 failures per workload as expected.

Cold fanout is the hardest boundary

The cold_fanout workload (adversarial random access) exposes the fundamental limitation: no prefetch policy can save a workload with no predictability. FixedWindow achieves 98.4% accuracy here because window size matches the fanout pattern — but at 1.5x waste rate due to double-promotion.

DRAM capacity is the primary cost driver

Sweeps from 16 MB to 512 MB DRAM show: structured KV workloads benefit from 64-128 MB, random old-context needs 256-512 MB, and cold fan-out fails at all capacities. The deployment model projects ~49% TCO savings for enterprise RAG vs all-DRAM baseline.
